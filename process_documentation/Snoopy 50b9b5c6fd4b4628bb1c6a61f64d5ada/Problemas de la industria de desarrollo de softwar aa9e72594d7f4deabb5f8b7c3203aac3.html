Problemas de la industria de desarrollo de software entorno a documentación y buen código (pre-finalistas)

Según chatGPT estos son algunos de los mayores problemas cuando se trata de hacer código bien documentado.

  1. Falta de tiempo para hacer buen código.
  1. Los requerimientos del código van cambiando con el tiempo, por lo que es difícil mantenerlos actualizados.
  1. Falta de guias estándar para documentar, la inconsistencia con las prácticas de documentación hace complejo que se entienda y mantenga el código a través de distintas plataformas o equipos. Sin lineamientos establecidos, estilo de documentación, formatos, el nivel de detalle puede variar mucho.
  1. Los desarrolladores podrían no tener suficiente entrenaiento o experiencia escribiendo documentación.
  1. Balancear detalle y legibilidad, la documentación debería tener suficiente detalle técnico para los desarrolladores y seguir siendo legible y entendible para distintas partes interesadas.

¿Qué documentaciones hay?

Existen varias guías estándar y prácticas de documentación ampliamente utilizadas para documentar un proyecto de software. A continuación, se presentan algunos ejemplos:

  1. Especificación de Requisitos de Software (SRS): Este documento captura los requisitos funcionales y no funcionales del software. Describe el alcance del proyecto, historias de usuario, casos de uso, restricciones del sistema y otros detalles importantes.
  1. Documentación de la API: Al desarrollar software con una interfaz de programación de aplicaciones (API), es crucial documentar la API. Algunos estándares comúnmente utilizados para la documentación de API incluyen OpenAPI (anteriormente conocido como Swagger), RAML (RESTful API Modeling Language) y API Blueprint.
  1. Manuales y Guías de Usuario: La documentación para usuarios proporciona instrucciones sobre cómo instalar, configurar y utilizar el software. Incluye manuales de usuario, guías de usuario, preguntas frecuentes y cualquier otro material destinado a ayudar a los usuarios finales a comprender y utilizar el software de manera efectiva.
  1. Documentación de Código: La documentación de código explica el funcionamiento interno del código fuente del software. Por lo general, incluye comentarios dentro del código fuente que brindan explicaciones de funciones, clases, módulos y su propósito, descripciones de entrada/salida y cualquier restricción o suposición relevante.
  1. Documentación de Arquitectura y Diseño: Esta documentación se centra en la arquitectura general y las decisiones de diseño del software. Puede incluir diagramas arquitectónicos, diagramas de flujo del sistema, diagramas de interacción de componentes, esquemas de bases de datos y otros artefactos que ayudan a los desarrolladores a comprender la estructura y las relaciones dentro del software.
  1. Notas de Lanzamiento: Las notas de lanzamiento documentan los cambios, correcciones de errores y mejoras introducidas en cada lanzamiento del software. Sirven como canal de comunicación entre los equipos de desarrollo y los usuarios finales, resaltando las nuevas características, problemas conocidos e instrucciones de actualización.
  1. Documentación del Sistema de Control de Versiones: Aunque no es específica de un solo proyecto, documentar las prácticas y convenciones del sistema de control de versiones ayuda a garantizar que los desarrolladores comprendan cómo administrar los repositorios de código de manera efectiva. Esto incluye pautas para ramificar, fusionar, confirmar código y resolver conflictos.
  1. Documentación de Pruebas: La documentación relacionada con las pruebas incluye planes de prueba, casos de prueba, scripts de prueba y resultados de pruebas. Describe la estrategia de pruebas, la cobertura de pruebas y proporciona instrucciones para ejecutar y validar el comportamiento del software.

Es importante tener en cuenta que las necesidades específicas de documentación pueden variar según el proyecto, su complejidad y los estándares de la industria que se sigan. La elección de las guías y prácticas de documentación adecuadas debe basarse en los requisitos del proyecto y en las necesidades del equipo de desarrollo y los usuarios finales.

¿Por qué es importante?

  1. La transferencia de conocimiento es crucial cuando se están haciendo proyectos, ya que contribuye a que otros puedan entenderlo y mantenerlo efectivamente.
  1. La calidad del código es esencial para el mantenimiento a largo plazo del software. Una documentación adecuada permite a los desarrolladores comprender cómo funciona el código, lo que facilita la identificación y corrección de errores, así como la implementación de nuevas características.
  1. Facilita la colaboración: Una documentación clara y completa fomenta la colaboración entre miembros del equipo de desarrollo, ya que todos pueden entender y trabajar con partes específicas del código o componentes del software sin depender únicamente de la experiencia de un solo desarrollador.
  1. Reducción de tiempos de resolución de problemas: Una buena documentación puede acelerar la resolución de problemas, ya que proporciona información sobre cómo se diseñó y desarrolló el software. Esto es especialmente útil cuando surgen problemas en el software en producción.
  1. Mejora la calidad del software: La documentación ayuda a los desarrolladores a seguir mejores prácticas de programación, ya que pueden comprender el propósito y la funcionalidad de diferentes partes del código. Esto contribuye a la creación de un código más limpio y eficiente.
  1. Mejora la experiencia del usuario final: La documentación de usuario final asegura que los usuarios puedan utilizar el software de manera efectiva, lo que mejora su satisfacción y reduce la necesidad de soporte técnico.
